ClickHouse极速上手

# 一、ClickHouse

# 1、什么是ClickHouse？

ClickHouse是一种开源的分布式列式存储数据库管理系统，用于处理海量数据。它旨在提供高性能、可靠性和快速查询速度。与传统的行式数据库系统不同，ClickHouse以列的方式存储数据，这使得它在大数据量下能够进行快速的复杂查询和聚合操作。

ClickHouse最初是由俄罗斯搜索引擎公司Yandex于2016年开发的，目前已经成为了许多企业和组织中处理大数据的首选工具之一。ClickHouse支持SQL查询语言，并具有高并发、分布式、水平扩展和可扩展性等特点，可以用于各种大数据场景，如实时分析、日志处理、统计分析等。

国外查看数据库排名的网站：DB-Engines Ranking - popularity ranking of database management systems (opens new window)

# 2、ClickHouse适用的场景

ClickHouse适用于需要处理大量数据的各种场景，包括但不限于以下几种：

  1. 实时分析：ClickHouse支持高性能的实时分析，可以快速查询海量数据并生成实时报告和分析结果。
  2. 数据仓库：ClickHouse作为数据仓库系统，可以存储大量数据、支持复杂查询和聚合操作，并提供高速的数据检索能力。
  3. 日志处理：ClickHouse可以非常方便地处理日志数据，并提供各种数据类型的支持，如字符串、数字、日期等。
  4. 统计分析：ClickHouse具有强大的聚合功能，可用于求和、平均值、最小值、最大值等统计分析操作。
  5. 时序数据：ClickHouse针对时序数据设计了专门的时间序列引擎，可以存储和查询大量的时序数据。

主要特征：

  • 读多写少，数量大
  • CPU占用率高，适用于后端数仓的建设的单独部署

总之，如果您需要处理大量数据、进行复杂查询和聚合操作、需要高性能和可扩展性，那么ClickHouse将是非常适合的选择。

# 二、ClickHouse环境安装

# 1、本地部署

clickhouse本地部署的方式有很多种，例如：DEB安装包,RPM安装包,Tgz安装包,Docker安装包等。

通常在学习的环境下最好是使用：

  • Tgz安装包（效率低，方便理解底层）
  • Docker安装包（效率高，方便测试）

但是在开发过程中最好方式是下载源码自行编译。

安装包下载地址：https://packages.clickhouse.com/tgz/stable/ (opens new window)

下面是下载最新稳定版的安装脚本：直接复制粘贴到服务器命令行中即可，里面使用正则匹配了最新的稳定版，并将其下载进行安装。

LATEST_VERSION=$(curl -s https://packages.clickhouse.com/tgz/stable/ | \
    grep -Eo '[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+' | sort -V -r | head -n 1)
export LATEST_VERSION

case $(uname -m) in
  x86_64) ARCH=amd64 ;;
  aarch64) ARCH=arm64 ;;
  *) echo "Unknown architecture $(uname -m)"; exit 1 ;;
esac

for PKG in clickhouse-common-static clickhouse-common-static-dbg clickhouse-server clickhouse-client
do
  curl -fO "https://packages.clickhouse.com/tgz/stable/$PKG-$LATEST_VERSION-${ARCH}.tgz" \
    || curl -fO "https://packages.clickhouse.com/tgz/stable/$PKG-$LATEST_VERSION.tgz"
done

tar -xzvf "clickhouse-common-static-$LATEST_VERSION-${ARCH}.tgz" \
  || tar -xzvf "clickhouse-common-static-$LATEST_VERSION.tgz"
sudo "clickhouse-common-static-$LATEST_VERSION/install/doinst.sh"

tar -xzvf "clickhouse-common-static-dbg-$LATEST_VERSION-${ARCH}.tgz" \
  || tar -xzvf "clickhouse-common-static-dbg-$LATEST_VERSION.tgz"
sudo "clickhouse-common-static-dbg-$LATEST_VERSION/install/doinst.sh"

tar -xzvf "clickhouse-server-$LATEST_VERSION-${ARCH}.tgz" \
  || tar -xzvf "clickhouse-server-$LATEST_VERSION.tgz"
sudo "clickhouse-server-$LATEST_VERSION/install/doinst.sh" configure
sudo /etc/init.d/clickhouse-server start

tar -xzvf "clickhouse-client-$LATEST_VERSION-${ARCH}.tgz" \
  || tar -xzvf "clickhouse-client-$LATEST_VERSION.tgz"
sudo "clickhouse-client-$LATEST_VERSION/install/doinst.sh"

# 三、ClickHouse的使用

ClickHouse作为数据库，对普通的整删改查的操作都是支持的。但是他针对数仓的使用场景，有着非常多的高级特性。针对这些高级特性的掌握程度将直接影响到clickhouse的使用效率。不同的SQL写法在Clickhouse上很容易体现出非常大的差别。

# 1. 库操作

-- 创建库
create database my_database; --  [ENGINE = Atomic]
-- 查看数据库
show databases;
-- 查看数据库详情 默认的引擎为：ENGINE = Atomic 
-- 该类型由clickhouse自己管理，每个数据库对应/var/lib/data/目录下的一个子目录
-- 每个表会分配一个唯一的UUUID.
show create database my_database; 

-- 数据库引擎的分类： 默认：Atomic
1. -- MaterializeMySQL：包含MySQL中所有的表，以及这些表中的所有数据，ClickHouse服务器作为MySQL副本工作。它读取binlog并执行DDL和DML查询。[experimental],目前这个还是实验性操作，过程中还会遇到很多细节的问题]
CREATE DATABASE [IF NOT EXISTS] db_name [ON CLUSTER cluster]
ENGINE = MaterializeMySQL('host:port', ['database' | database], 'user', 'password') [SETTINGS ...]

2. -- MySQL：MySQL引擎用于将远程的MySQL服务器中的表映射到ClickHouse中，并允许您对表进行INSERT和SELECT查询，以方便您在ClickHouse与MySQL之间进行数据交换，MySQL数据库引擎会将对其的查询转换为MySQL语法并发送到MySQL服务器中，因此您可以执行诸如SHOW TABLES或SHOW CREATE TABLE之类的操作。(性能不高，需要传输网络，交互)
CREATE DATABASE [IF NOT EXISTS] db_name [ON CLUSTER cluster]
ENGINE = MySQL('host:port', ['database' | database], 'user', 'password')

3. SQLite,PostgreSQL 等等实验性操作等

这些实验性的操作，可以等后续测试稳定再进行投入使用，其实在库的操作上，使用默认的引擎和Mysql引擎已经满足大多数日常开发需求，后面更细化的操作也可以在表中进行。

# 类型的定义

数值型:  后面的 数字代表了占用的字节
	整型：严格区分大小写
		Int8（对应java的byte),
        Int16 (对应java的short),
        Int32 (对应java的int),
        Int64 (对应java的long),
        
	无符号整型：UInt8, UInt16, UInt32, UInt64。
	
	浮点类型：Float32（float), Float64（double)
	注意：官方建议使用整形来纯粹数据，因为使用浮点数会有精度丢失的问题。
	
	有符号的浮点数：Decimal类型
	    Decimal32(s)
	    Decimal64(s)
	    Decimal128(s) #不建议用这个128，现代Cpu不支持128位数字，是由操作模拟的，运算数度明显略慢.
	    s是表示小数点后的数字位数.

字符串型: String, FixedString。
枚举类型: Enum8，Enum32。
复合类型: Array, Tuple。
时间日期型: Date, DateTime, DateTime64。
特殊类型: UUID, IPv4, IPv6, Nullable。

其中，数值型用于存储数字数据，支持整型和浮点型；时间日期型用于存储日期和时间数据；字符串型用于存储字符串数据，包括定长和变长字符串；复合类型用于存储数组和元组数据；枚举类型用于存储枚举数据；特殊类型用于存储UUID、IPv4和IPv6地址和可空数据等。

# 表引擎(核心)

表引擎（即表的类型）决定了：数据的存储方式和位置，写到哪里以及从哪里读取数据，支持哪些查询以及如何支持，并发数据访问，索引的使用（如果存在），是否可以执行多线程请求，数据复制参数。

# 引擎分类

合并树家族：

  • VersionedCollapsingMergeTree：允许快速写入不断变化的对象状态。删除后台中的旧对象状态。 这显著降低了存储体积。
  • GraphiteMergeTree：数据进行瘦身及汇总。
  • AggregatingMergeTree：支持对数据进行预先聚合，适用于需要进行快速、经常性的聚合计算的情况。
  • CollapsingMergeTree：可以显著的降低存储量并提高 SELECT 查询效率。
  • 分区表：例如，指定按月分区
  • MergeTree：支持快速插入和排序，并且可以在大数据量的情况下高效地进行查询和聚合操作。
  • ReplacingMergeTree：适用于在后台清除重复的数据以节省空间，但是它不保证没有重复的数据出现。
  • SummingMergeTree：支持数据按照指定的主键进行汇总，适用于统计场景。
  • ....

日志：最小功能的轻量级引擎，快熟写入许多小表，然后并整体读取他们，通常用于练习

  • TinyLog
  • StripeLog
  • Log

集成：kafka，Mysql，ODBC，JDBC，HDFS，Hive等等

其他特定功能：

  • Distributed：分布式表引擎，用于将数据存储在多个ClickHouse节点上，以实现数据的水平扩展和容错能力
  • ....

常用的表引擎是MergeTree和Distributed。其中，MergeTree引擎的优势在于插入快、查询快，同时还具有较好的压缩比和占用小的磁盘空间。Distributed引擎则能够实现数据的水平扩展和容错能力，可用于存储数据较大、访问频繁的场景。

在进行选型时，需要根据实际业务需求和系统特点进行综合考虑。如果数据量较大，查询和聚合操作频繁，可考虑选择MergeTree引擎。如果需要实现数据的分布式存储和访问，须使用Distributed引擎。同时，还需考虑数据压缩比、磁盘空间占用等因素。

下列主要挑俩个常用的进行扩展讲解展示。

# MergeTree

特点：存储的数据按主键排序，可以使用分区，支持数据副本，支持数据采样。

CREATE TABLE [IF NOT EXISTS] [db.]table_name [ON CLUSTER cluster]
(
    name1 [type1] [DEFAULT|MATERIALIZED|ALIAS expr1] [TTL expr1],
    name2 [type2] [DEFAULT|MATERIALIZED|ALIAS expr2] [TTL expr2],
    ...
    INDEX index_name1 expr1 TYPE type1(...) GRANULARITY value1,
    INDEX index_name2 expr2 TYPE type2(...) GRANULARITY value2
) ENGINE = MergeTree()
ORDER BY expr
[PARTITION BY expr]
[PRIMARY KEY expr]
[SAMPLE BY expr]
[TTL expr [DELETE|TO DISK 'xxx'|TO VOLUME 'xxx'], ...]
[SETTINGS name=value, ...]

# ReplacingMergeTree

特点：会删除排序键值相同的重复项。

CREATE TABLE [IF NOT EXISTS] [db.]table_name [ON CLUSTER cluster]
(
    name1 [type1] [DEFAULT|MATERIALIZED|ALIAS expr1],
    name2 [type2] [DEFAULT|MATERIALIZED|ALIAS expr2],
    ...
) ENGINE = ReplacingMergeTree([ver])
[PARTITION BY expr]
[ORDER BY expr]
[SAMPLE BY expr]
[SETTINGS name=value, ...]
-- 解释
ver — 版本列。类型为 UInt*, Date 或 DateTime。可选参数。
在数据合并的时候，ReplacingMergeTree 从所有具有相同排序键的行中选择一行留下：
如果 ver 列未指定，保留最后一条。
如果 ver 列已指定，保留 ver 值最大的版本。

合并树家族其他分类的理解：本质是在合并的时候做了一些不同的操作，例如：ReplacingMergeTree合并的时候去重，SummingMergeTree根据相同的主键合并的时候对数值类型的列进行汇总。等等,合并过程中的一些列操作。

总结：真正的核心就是MergeTree。MergeTree一系列表引擎的底层都使用类型LSM树的方式提供数据快速读写功能。即新更新或者删除的数据并不会直接影响到原有的数据，会新开辟一块临时数据中，查询时会通过版本号查询最新的一条结果。新开辟的数据需要等后台数据合并时才能加入主数据中。数据的合并在某个不确定的时间点进行，也可以手动促发。

这一系列表引擎中，以MergeTree为核心，确定了整个数据的读写合并的机制，而扩展出来的一些列*MergeTree则大都是在数据合并的过程中定制数据合并的逻辑。例如：ReplacingMergeTree合并的时候去重，SummingMergeTree合并的时进行累加统计。AggregatingMergeTree也是对合并的逻辑进行定制。

具体其他引擎了解地址：https://clickhouse.com/docs/zh/engines/table-engines (opens new window)

# 四、集群机制

其实clickhouse的单机性能通常已经非常优秀了，底层的数据压缩效率是很高的。除此之外，clickhouse还提供集群的机制，主要作用有俩个，一是数据副本：将数据备份到另外的机器上用于保证高可用，二是分布式表，将一个表的数据分散在多个节点上保存。

# 数据副本

通过写入日志(元数据)到zookeeper来让协调副本，写入的机子是平等。数据副本需要经过网络传输是异步实现，所以会有延迟。对于Insert语句，只会等待一个副本写入成功后就返回。多个副本情况下，clickhouse是可能有丢失数据的。写入数据时，clickhouse只保证了单个数据块的写入是原子的，不能保证所有的数据写入是原子的。一个数据块的大小可以根据：max_insert_block_size=1048576进行分块。数据块的写入时会去重，同样的insert语句多次重复执行，数据库块只会执行一次。是为了放置优化重复插入或者网络波动等原因引起的数据重发。去重机制只对Replicated*MergeTree系列的表引擎有效，而普通的MergeTree是不带这个去重功能的。

# 五、配置优化

CPU使用率是对clickhouse服务进行监控的一个很重要的指标，一般CPU使用率达到50%左右会出现查询波动，达到70%会出现大范围的查询超时。所以通常建议clickhouse服务要单独进行部署。相比内存和磁盘其数据的压缩效率是非常高的，所以通常不会形成性能瓶颈。大部分的clickhouse的配置详细的默认值，大部分的场景下是最优化的。最关键的是users.xml中设置了单个任务使用内存的上限，设置的值也是简单粗暴的10G，可以通过增加<max_memory_usage>参数值和<max_block_size>参数值来增加缓存大小和每个块的最大内存限制。通过增加这些参数值，可以提高ClickHouse的并发处理能力，从而提升系统性能.可以通过配置<compression>参数来调整数据压缩比例。增加数据压缩比例可以减少磁盘IO负载，并且可以节省磁盘空间。

  • config.xml 配置系统运行的参数，服务端
  • user.xml 运行环境相关，俩者区分性不是特别明显

服务器配置 | ClickHouse Docs (opens new window)

  • 文件 user.xml中配置
  • session配置。例如：SET [settings]=[value]
  • 查询时指定。例如：--setting=value

比较有用的几个配置项：

  • background_pool_size 后台线程池的大小。默认是16，实际生产可以改为CPU个数的俩倍。
  • background_distributed_schedule_pool_size 分布式表发生执行后台任务的线程数。默认是16，如果表比较多，可以改为CPU个数的俩倍。
  • max_concurrent_ queries:最大并发请求数，默认值100。通常不建议调整，如果并发要求高，可以以50为阶段调整。
  • max_memory_usage:单次查询使用的最大内存大小。 默认100G。如果服务器的内存资源很丰富，可以适量调大- -点.
  • default_ session_ timeout :默认session断开时间，默认60秒。另外还有max_ session. _timeout. 最大session断开时间， 默认3600秒。需要控制客户端连接时长时可以定制。
  • http_port: http接口访问端口，默认8123. clickhouse最重要的端口。不光在HTTP接口中用，jdbc,obdc驱动也使用这个端口。与Grafana等监控平台集成也可以通过这个端口。
  • mysq_port:默认值9004. clickhouse在这个端口上, 将会假装成-个mysq|数据库。直接使用mysq|的jdbc驱动包就可以连接。
  • postgresql_port:默认值9005. clickhouse在这个端口上， 将会假装成一个postgresq|数据库。
  • interserver_http. port:默认9009。 clickhouse内部通信的端口。
  • tcp_port:通过TCP协议与客户端通信的端口。

这几个端口，如果在生产环境是需要时，可以配置安全组。

# 高性能查询优化

  1. 选择合适的表引擎：MergeTree是最为常用的表引擎，但不意味着所有的场景都适合。

  2. 建表不要使用Nullable：会拖累性能，也带来一些类型转换的不便。

  3. 合适的规划分区和索引：建议按天分区，如果自行分区，但分区数量最好不要超过一百万行。

  4. 数据变更优化：对数据的增删改查都会产生新的临时分区，数据变更不宜太频繁，建议一秒钟左右发起一次写入操作，每次写入数据量保持在2W~5W之间。具体需要根据服务器性能而确定。

  5. 使用Prewhere代替where：用法跟where一样，只支持*MergeTree系列引擎。它会先读取指令列数据，然后判断数过滤，等待过滤完后再读取select声明的字段列来补全其他属性。相比，where语句会读取整行各列的数据，再进行过滤，IO性能明细下降。通常情况下，默认就会变更为Prewhere。

  6. 指定列和分区：避免使用select * ,过滤的时候where 分区 过滤

  7. 避免构建虚拟列：尽量使用clickhouse中的表已有的列，不要将计算结果构建成不存在的虚拟列，非常消耗性能和计算资源。通常情况下，都可以在数据进入clickhouse之前进行处理。

  8. 用IN代替JOIN：如果非要用JOIN，也要尽量把大表写在最前面，小表写在最后面。

    select a.CounterID from datasets.hits_ v1 a where a. CounterID in (select CounterID from datasets.visits_ v1) limit 100;
-- 结果100 rows in set. Elapsed: 0.047 sec. Processed 262.02 thousand rows, 1.05 MB (5.55 million rows/s., 22.21 MB/s.)

select a.CounterID from datasets.hits v1 a left join datasets.visits v1 b on a.CounterID=b.CounterID limit 100;
-- 结果100 rows in set. Elapsed: 0.156 sec. Processed 1.68 million rows, 6.71 MB (10.74 million rows/s., 42.98 MB/s.)

# 七、生产常见问题

数据一致性问题

  • 尽量避免使用分布式表，因为其没有很好的写入失败重试功能，如果非要用可以设置一次写入数据少一些，频率高一些。

  • 多副本造成的数据一致性问题，只能保证最终一致性，不能保证强一致性。使用手动促发合并或者在表中加入乐观锁标志为最新的数据。

多副本表，尽量固定写入的节点：固定一个写入节点有助于在节点宕机后，可以从其他的节点清楚定位到其写入的范围，不会出现参差不齐的情况。数据的恢复也很简单，将主节点的metadata和data全部清空，将副本节点的metadata和data目录拷贝过来，然后重启数据库即可。

Zookeeper数据丢失导致副本表无法启用：如果配置了副本表或者分布式表，导致丢失了一部分的副本表信息，而clickhouse的metadata元数据依然存在zookeeper的信息，就会导致启动报错。这时需要手动重建zookeeper里的信息。手动恢复不太可能，这时可以移除问题节点上的metadata中的对应表结构文件，将click house服务先启动起来，然后重新创建表。建表语句可以在metadata中的表结构文件中获取，也可以在其他的节点上执行show create table 语句获取。

其他常见问题排查参考：https://help.aliyun.com/document_detail/162815.html (opens new window)
