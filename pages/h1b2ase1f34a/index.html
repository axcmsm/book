hbase笔记

# 简介

HBase 是一种分布式、可扩展、支持海量数据存储的 NoSQL 数据库。

官网:https://hbase.apache.org/ (opens new window)

Hive和Hbase比较：

  • Hive是分析清洗数据的,做数据仓库。
  • habase是用来存储数据的,面向列族存储的非关系型数据库,存储结构化和非结构化的数据,适用于单表查询,基于hdfs,延迟低,接入在线业务适用(用户画像）(类似 es、redis)

数据模型:可分为逻辑结构(映射成表)、和物理结构(k,v)

逻辑结构：

物理存储结构：实际存储数据的结构

解释：

1）Name Space 命名空间
类似与数据库的概念,自带了hbase和default,一个是存储内置表，一个是默认使用的命名空间
2）Region
类似于表的概念,创建声明即可,字段可动态按需指定。
3）Row 行
每行数据都由一个 RowKey 和多个 Column（列）组成。rowkey按照字典序进行排序存储的。
(查询会根据rowkey进行检索,也有其他分析框架结合rowkey二次索引,提高查询效率)
4）Column 列
每个列由Column Family(列族)和 Column Qualifier（列限定符）进行限定.
5）Time Stamp 时间戳
用于标识数据的不同版本（version）,是很重要的标识
6）Cell   数据
由{rowkey, column Family：column Qualifier, time Stamp} 唯一确定的单元,没有类型，全部为字节码形式存储

# 架构

基础架构:

解释：

1）Region Server
Region Server为Region的管理者,实现类HRegionServer,
对于数据的操作：get, put, delete；
对于Region的操作：splitRegion(分)、compactRegion(合)。
2）Master
Master 是所有RegionServer的管理者,其实现类为HMaster;
对于表的操作：create, delete, alter
分配 regions到每个RegionServer，监控每个 RegionServer的状态，负载均衡和故障转移
3）Zookeeper
用作做Master的高可用、RegionServer 的监控、元数据的入口以及
集群配置的维护等工作
4）HDFS 
存储数据的地方

详细架构:

解释

1）StoreFile
实际存储文件,以HFile的形式存储在HDFS上,StoreFile在store中都是有序的
2）MemStore
写缓存,数据线存储到MemStore排好序后,等到达刷写时机再刷写到hfile，每次刷写产生一个新的 HFile
3）WAL
用来做写入预写日志操作,因为数据写入内存会又很高概率丢失,可通过日志重建。

写流程

解释：

1）Client先访问zookeeper，获取 hbase:meta 表位于哪个 Region Server。
2）访问对应的 Region Server，获取 hbase:meta 表，根据读请求的 namespace:table/rowkey，
查询出目标数据位于哪个 Region Server 中的哪个 Region 中。并将该 table 的 region 信息以
及 meta 表的位置信息缓存在客户端的 meta cache，方便下次访问。
3）与目标 Region Server 进行通讯；
4）将数据顺序写入（追加）到 WAL；
5）将数据写入对应的 MemStore，数据会在 MemStore 进行排序； 6）向客户端发送 ack；
7）等达到 MemStore 的刷写时机后，将数据刷写到 HFile。

刷写流程 :MemStore Flush

解释：

# MemStore 刷写时机
1.当某个 memstroe 的大小达到了 hbase.hregion.memstore.flush.size（默认值 128M）
其所在 region 的所有 memstore 都会刷写。当 memstore 的大小达到了
hbase.hregion.memstore.flush.size（默认值 128M）* hbase.hregion.memstore.block.multiplier
（默认值 4）时，会阻止继续往该 memstore 写数据。
2.当 region server 中 memstore 的总大小达到
java_heapsize*hbase.regionserver.global.memstore.size（默认值 0.4） *hbase.regionserver.global.memstore.size.lower.limit（默认值 0.95），
region 会按照其所有 memstore 的大小顺序（由大到小）依次进行刷写。直到 region server
中所有 memstore 的总大小减小到上述值以下。 当 region server 中 memstore 的总大小达到
java_heapsize*hbase.regionserver.global.memstore.size（默认值 0.4）
时，会阻止继续往所有的 memstore 写数据。
3. 到达自动刷写的时间，也会触发 memstore flush。自动刷新的时间间隔由该属性进行
配置 hbase.regionserver.optionalcacheflushinterval（默认 1 小时）。
4.当 WAL 文件的数量超过 hbase.regionserver.max.logs，region 会按照时间顺序依次进
行刷写，直到 WAL 文件数量减小到 hbase.regionserver.max.log 以下（该属性名已经废弃，
现无需手动设置，最大值为 32）。

读流程

解释

1）Client 先访问 zookeeper，获取 hbase:meta 表位于哪个 Region Server。
2）访问对应的 Region Server，获取 hbase:meta 表，根据读请求的
namespace:table/rowkey，查询出目标数据位于哪个 Region Server 中的哪个
Region中。并将该table的region信息以及meta表的位置信息缓存在客户端的meta cache方便下次访问。
3）与目标 Region Server 进行通讯；
4）分别在 Block Cache（读缓存），MemStore 和 Store File（HFile）中查询目标数据，并将
查到的所有数据进行合并。此处所有数据是指同一条数据的不同版本（time stamp）或者不
同的类型（Put/Delete）。
5） 将从文件中查询到的数据块（Block，HFile 数据存储单元，默认大小为 64KB）缓存到
Block Cache。 6）将合并后的最终结果返回给客户端。

合并流程：StoreFile Compaction

解释：

由于memstore每次刷写都会生成一个新的HFile，同一个字段的不同版本（timestamp）和
不同类型（Put/Delete）有可能会分布在不同的 HFile 中,因此查询时需要遍历所有的 HFile
为了减少 HFile 的个数,及清理掉过期和删除的数据，会进行StoreFile Compaction合并

合并分为
  - Minor Compaction 小合并(不会删除数据)
  - Major Compaction 大合并(会删除数据,需要等几分钟)

切分流程

解释：

数据的不断写入,Region 会自动进行拆分,一道俩断,都位于当前的 Region Server,
为了负载均衡,有可能会将某个 Region 转移给其他的 Region Server,

Region Split 时机
 - 当1个region中的某个Store下所有StoreFile的总大小超过hbase.hregion.max.filesize，
 	该 Region 就会进行拆分（0.94 版本之前）
 - 当1个region中的某个Store下所有StoreFile的总大小超过Min(R^2 *
    "hbase.hregion.memstore.flush.size",hbase.hregion.max.filesize")，
    该Region 就会进行拆分，其中 R 为当前 Region Server 中属于该 Table 的个数（0.94 版本之后）

   truncate 'student'#清空表前要disable下线表
10.删除表
   disable 'student' #下线表
   drop 'student'
11.变更表信息
   alter 'student',{NAME=>'info',VERSIONS=>3}

package com.axcmsm.test;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.hbase.*;
import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.util.Bytes;
import java.io.IOException;
import java.util.Iterator;

/*
 * DDL：
 * 1判断表是否存在
 * 1.创建表
 * 3.创建命名空间
 * 4.删除表
 *
 * DML:
 * 1.插入数据
 * 2.查数据(get)
 * 3.查数据(scan)
 * 4.删除数据
 *
 *
 * */
public class TestApi {
    private static Connection connection = null;
    private static Admin admin = null;

    static {
        //        HBaseConfiguration conf = new HBaseConfiguration();
        Configuration conf = HBaseConfiguration.create();
        //连接zk
        conf.set("hbase.zookeeper.quorum", "hadoop111,hadoop112,hadoop113");
//        HBaseAdmin admin = new HBaseAdmin(conf);
        try {
//           创建连接对象
            connection = ConnectionFactory.createConnection(conf);
        } catch (IOException e) {
            e.printStackTrace();
        }
        try {
//            创建admin对象
            admin = connection.getAdmin();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }


    //    1判断表是否存在
    public static boolean isTablieExit(String tableName) throws IOException {
        boolean flag = admin.tableExists(TableName.valueOf(tableName));
        return flag;
    }

    //关闭方法
    public static void close() {
        if (admin != null) {
            try {
                admin.close();
            } catch (IOException e) {
                e.printStackTrace();
            }
        }
        if (connection != null) {
            try {
                connection.close();
            } catch (IOException e) {
                e.printStackTrace();
            }
        }
    }

//    创建表
    public static void createTable(String tableName,String... cfs) throws IOException {
        if (cfs.length<=0){
            System.out.println("请设置列族信息");
            return;
        }
        if (isTablieExit(tableName)){
            System.out.println("表已经存在");
            return;
        }
        HTableDescriptor table = new HTableDescriptor(TableName.valueOf(tableName));
        //循环添加列族
        for (String cf : cfs) {
            //创建列族描述器
            HColumnDescriptor family = new HColumnDescriptor(cf);
            table.addFamily(family);//添加列族信息
        }
        admin.createTable(table);
    }

    //删除表
    public static void dropTable(String tableName) throws IOException {
        if (!isTablieExit(tableName)) {
            System.out.println("不存在");
            return;
        }
        //下线
        admin.disableTable(TableName.valueOf(tableName));
        //删除表
        admin.deleteTable(TableName.valueOf(tableName));

    }
    //创建命名空间
    public static void createNameSpace(String ns){
        try {
            admin.createNamespace(NamespaceDescriptor.create(ns).build());
        }catch (NamespaceExistException e) {
            System.out.println(e+"命名空间已存在！！！！！！！！！！");
        }
        catch (IOException e) {
            e.printStackTrace();
        }
    }

    //插入数据
    public static void putData(String tableName,String rowKey,String cf,String cn,String value) throws IOException {
        Table table = connection.getTable(TableName.valueOf(tableName));
        Put put = new Put(Bytes.toBytes(rowKey));//传入rowkey
        put.addColumn(Bytes.toBytes(cf), Bytes.toBytes(cn), Bytes.toBytes(value));
        table.put(put);//插入数据
        //关闭表连接
        table.close();
    }

    //获取数据
    public static void getData(String tableName,String rowKey,String cf,String cn) throws IOException {
        Table table = connection.getTable(TableName.valueOf(tableName));
        //指定rowkey
        Get get = new Get(Bytes.toBytes(rowKey));
        //        指定列族  (不指定默认按row获取)
//        get.addFamily(Bytes.toBytes(cf));
        //指定列族和列
        get.addColumn(Bytes.toBytes(cf),Bytes.toBytes(cn));
        //获取数据的半本数
        get.setMaxVersions(5);//最大半本数
        //  解析
        Result result = table.get(get);
        for (Cell cell : result.rawCells()) {
            System.out.println("CF:" + Bytes.toString(CellUtil.cloneFamily(cell)) +
                    ", CN: " + Bytes.toString(CellUtil.cloneQualifier(cell))+
                    ", value: "+Bytes.toString(CellUtil.cloneValue(cell)));
        }
        table.close();
    }

    //获取数据
    public static void scanData(String tableName) throws IOException {
        Table table = connection.getTable(TableName.valueOf(tableName));

        //空参全表  可以传入rowkey 和列族 还有过滤器
        ResultScanner scanner = table.getScanner(new Scan(Bytes.toBytes("1001")));
        for (Result result : scanner) {
            //解析result
            for (Cell cell : result.rawCells()) {
                System.out.println("Rk:"+Bytes.toString(CellUtil.cloneRow(cell))+
                        ",CF:" + Bytes.toString(CellUtil.cloneFamily(cell)) +
                        ", CN: " + Bytes.toString(CellUtil.cloneQualifier(cell))+
                        ", value: "+Bytes.toString(CellUtil.cloneValue(cell)));
            }
        }
        //关闭连接
        table.close();
    }

    //删除数据
    public static void deleteDate(String tableName,String rowkey,String cf,String cn) throws IOException {
        Table table = connection.getTable(TableName.valueOf(tableName));

        // 只传入rk 就相当于deleteall
        Delete delete = new Delete(Bytes.toBytes(rowkey));
        //指定列族、列删除新版本   慎用 你删除新版本旧版本又出来了
//        delete.addColumn(Bytes.toBytes(cf),Bytes.toBytes(cn));
        //指定列族、列删除全部版本
//        delete.addColumns(Bytes.toBytes(cf),Bytes.toBytes(cn));
        //指定列族
        delete.addFamily(Bytes.toBytes(cf));
        table.delete(delete);
        table.close();

    }

    public static void main(String[] args) throws IOException {

//        dropTable("0408:stu5");

////        createNameSpace("0408");

////        插入数据测试
//        putData("0408:stu5","1001","info1","name","lisi");
//        putData("0408:stu5","1001","info1","age","12");
//        //获取单行数据
//        getData("0408:stu5","1001","info1","name");
        //获取全表数据
        scanData("0408:stu5");
        System.out.println("------------------");
//        System.out.println(isTablieExit("0408:stu5"));
//        System.out.println("------------------");

    }

内存优化:不建议分配非常大的堆内存,跟刷写有关

基础优化：

  • 允许在 HDFS 的文件中追加内容

    hdfs-site.xml、hbase-site.xml
属性：dfs.support.append
解释：开启 HDFS 追加同步，可以优秀的配合 HBase 的数据同步和持久化。默认值为 true。
    1
    2
    3

  • 优化 DataNode 允许的最大文件打开数

    hdfs-site.xml
属性：dfs.datanode.max.transfer.threads
解释：HBase 一般都会同一时间操作大量的文件，根据集群的数量和规模以及数据动作，
设置为 4096 或者更高。默认值：4096
    1
    2
    3
    4

  • 优化延迟高的数据操作的等待时间

    hdfs-site.xml
属性：dfs.image.transfer.timeout
解释：如果对于某一次数据操作来讲，延迟非常高，socket 需要等待更长的时间，建议把
该值设置为更大的值（默认 60000 毫秒），以确保 socket 不会被 timeout 掉。
    1
    2
    3
    4

  • 优化数据的写入效率

    mapred-site.xml
属性：
mapreduce.map.output.compress
mapreduce.map.output.compress.codec
解释：开启这两个数据可以大大提高文件的写入效率，减少写入时间。第一个属性值修改为
true，第二个属性值修改为：org.apache.hadoop.io.compress.GzipCodec 或者其
他压缩方式。
    1
    2
    3
    4
    5
    6
    7

  • 设置 RPC 监听数量

    hbase-site.xml
属性：Hbase.regionserver.handler.count
解释：默认值为 30，用于指定 RPC 监听的数量，可以根据客户端的请求数进行调整，读写
请求较多时，增加此值。
    1
    2
    3
    4

  • 优化 HStore 文件大小

    hbase-site.xml
属性：hbase.hregion.max.filesize
解释：默认值 10737418240（10GB），如果需要运行 HBase 的 MR 任务，可以减小此值，
因为一个 region 对应一个 map 任务，如果单个 region 过大，会导致 map 任务执行时间
过长。该值的意思就是，如果 HFile 的大小达到这个数值，则这个 region 会被切分为两
个 Hfile。
    1
    2
    3
    4
    5
    6

  • 优化 HBase 客户端缓存

    hbase-site.xml
属性：hbase.client.write.buffer
解释：用于指定 Hbase 客户端缓存，增大该值可以减少 RPC 调用次数，但是会消耗更多内
存，反之则反之。一般我们需要设定一定的缓存大小，以达到减少 RPC 次数的目的。
    1
    2
    3
    4

  • 指定 scan.next 扫描 HBase 所获取的行数

    hbase-site.xml
属性：hbase.client.scanner.caching
解释：用于指定 scan.next 方法获取的默认行数，值越大，消耗内存越大
    1

    当MemStore 达到阈值,Memstore中的数据Flush进Storefile,compact机制则是把flush
出来的小文件合并成大的 Storefile 文件。split 则是当 Region 达到阈值，会把过大的 Region一分为二
    1

128M 就是 Memstore 的默认阈值

flush该HRegion的所有memstore。RegionServer的flush是通过

产生大量积压请求时，可能会导致内存陡增，最坏的情况是触发 OOM
